# ADR-011: Three-Tier Matching Architecture

**Status**: Accepted
**Date**: 2026-03-19
**Sprint**: 8

## Context

Sprint 7 used a single matching tier (`orchestrator/victory_matcher.py`) that scored all Tenex
wins on the same scale and produced `VictoryMatch` records with a three-label enum
(DIRECT_MATCH / CALIBRATION_MATCH / ADJACENT_MATCH). Eval scores showed `tier_classification`
at 3.67, below the 3.80 threshold. The core problem: recommended use cases could not distinguish
between "Tenex built this exact thing last year" and "the industry is experimenting with this."
Both looked the same to the use case generator and to the client reading the report.

Sai directed: delivered solutions are the core value prop. The system must separate what Tenex
has proven from what it can adapt and from what the market is doing -- with clear data
separation, independent ingestion, and distinct UI presentation for each category.

## Decision

Two separate ChromaDB collections replace the single `ai_solutions` collection:
- `tenex_delivered` -- Library A: Tenex-delivered engagements. Fed by `rag/ingest_solution.py`.
- `industry_cases` -- Library B: external market intelligence. Fed by `rag/ingest_industry_case.py`.

`orchestrator/matching_layer.py` is the only code that reads both collections and bridges them
to the analysis pipeline. No analysis agent reads either library directly.

The matching layer produces three typed output tracks from a single `match()` call:

| Track | Source | Confidence band | Threshold |
|-------|--------|-----------------|-----------|
| DELIVERED | Library A | 0.80-0.95 | score >= 0.75 |
| ADAPTATION | Library A | 0.55-0.79 | score 0.45-0.74 |
| AMBITIOUS | Library B | 0.40-0.65 | score >= 0.30 |

A Library A record appears in at most one track. If a record scores >= 0.75 it is DELIVERED,
not ADAPTATION. The higher-scoring track always wins.

`MatchResult` replaces `VictoryMatch` as the canonical output type. `VictoryMatch` is kept as
a deprecated alias in `orchestrator/schemas.py` for backwards compatibility.

Each tier gets its own synthesis prompt:
- `prompts/tier1_delivered.md` -- cites exact win_id, proven metrics verbatim
- `prompts/tier2_adaptation.md` -- cites base win_id, explains gap, applies discount factor
- `prompts/tier3_ambitious.md` -- cites external company or published source, no win-NNN refs

The use case generator makes three separate model calls per pipeline run, one per non-empty tier.

## Consequences

- Match evidence is traceable: DELIVERED traces to a Tenex win ID, AMBITIOUS traces to an
  external source. Clients can see the difference.
- Three model calls per pipeline run in use case generation. Calls are parallelizable using
  the same ThreadPoolExecutor pattern as the report writer.
- Two separate ingestion CLIs mean the knowledge base grows without touching analysis code.
- Per-tier eval rubrics (`evals/rubrics/match_quality.yaml`) are now needed to measure
  whether the matching layer assigned records to the correct tier.
- UI must render three visually distinct sections, not just tier badges on cards.
- `rag/ingest.py` `ensure_seeds_loaded()` must call both ingestion paths on startup.
- MockStore must support a `collection` parameter to return from the correct fixture file
  in dry-run mode.

## Alternatives considered

1. Single collection, three tiers by score band only -- rejected because Tenex wins and
   industry case studies have different fields, embedding strategies, and evidence requirements.
   Mixing them in one collection degrades retrieval quality.

2. Keep `VictoryMatch` schema, add a `source_library` field -- rejected because the existing
   schema does not carry the per-tier evidence fields (proven_metrics, adaptation_notes,
   industry_examples) that the synthesis prompts require.

3. Three separate matching functions, one per tier -- rejected because the single `match()`
   function enforces the "one record, one tier" invariant cleanly. Separate functions would
   require the caller to deduplicate.
