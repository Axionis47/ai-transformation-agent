# ADR-010: Cloud Run as Production Deploy Target with Zero-Secrets-in-Image Policy

**Status:** Accepted
**Date:** 2026-03-16

## Context

The service needs a production deploy target for the Tenex demo submission.
GCP Cloud Run was chosen in ADR-001 as part of the infrastructure stack.
This ADR formalises the Cloud Run configuration choices and the secrets handling policy.

## Decision

### Deploy target

Google Cloud Run (`us-central1`, project `plotpointe`, service `ai-transform-agent`) is the
sole production deploy target. All deploy operations go through `infra/deploy_target.py`.

### Build strategy

Images are built via **Cloud Build** (`gcloud builds submit`) so that:
- No Docker daemon is required on the deploying machine.
- Build logs are stored in GCP, not on developer machines.
- The image never leaves the GCP boundary before deployment.

### Container configuration

| Setting | Value | Reason |
|---------|-------|--------|
| Memory | 2 Gi | Chromadb + model client in-process |
| CPU | 1 vCPU | Adequate for synchronous request handling |
| Min instances | 0 | Zero cold-start cost when idle |
| Max instances | 3 | Cap spend during demo traffic spikes |
| Auth | `--allow-unauthenticated` | Public demo URL required for Tenex review |

### Zero-secrets-in-image policy

No API keys, credentials, or project-specific values are baked into the image.

- `.dockerignore` excludes `.env*`, `service_account.json`, `*.key`, `*.pem`.
- Runtime secrets are injected via `--set-env-vars` at deploy time.
- Secret Manager integration deferred to post-Tenex hardening sprint.

## Consequences

- Sai can trigger a deploy with a single `gcloud` command or via CI.
- Image is fully reproducible — no local state leaks into production.
- Cold-start latency on first request (~2 s) is acceptable for a demo.
- Secret rotation requires a new `gcloud run deploy` with updated `--set-env-vars`.
