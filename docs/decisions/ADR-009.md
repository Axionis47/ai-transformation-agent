# ADR-009: Parallel Report Section Generation via ThreadPoolExecutor

Status: Accepted
Date: 2026-03-16
Decider: Backend (Sprint 6)

---

## Context

The report generation stage (Stage 7) previously issued a single LLM call to
`ReportWriterAgent._run()` which returned all 5 report sections in one response.
On live Vertex AI calls, this is a long-tail latency bottleneck — the model must
generate ~2,000 tokens in one shot before the pipeline can complete.

The ticket asked us to parallelise this: fire 5 independent section-generation
calls concurrently so wall-clock latency approaches max(section_latency) rather
than sum(section_latencies).

---

## Decision

Use `concurrent.futures.ThreadPoolExecutor(max_workers=3)` to run up to 3 section
generations simultaneously. The fifth section queues behind one of the first three
completing, keeping Vertex AI concurrency at or below 3 — consistent with typical
per-project quota limits.

Key decisions:

1. **ThreadPoolExecutor over asyncio** — `run_pipeline` is a synchronous function.
   Introducing `asyncio.run()` inside a sync call creates event-loop complications
   and adds cognitive overhead with no measurable benefit. ThreadPoolExecutor is
   already imported in `pipeline.py` for `_run_with_timeout`; reusing it is
   consistent and simple.

2. **`generate_section(section, analysis)` on `ReportWriterAgent`** — Rather than
   modifying `_run`, we add a new public method. `_run` is kept as fallback for
   the single-call path. `generate_section` in dry-run mode reads the section from
   the existing fixture, so the parallel path is exercised in tests with zero real
   API calls.

3. **Section failures are non-fatal** — If an individual section fails (model error,
   timeout), the key is set to `None` and the pipeline continues. A partial report
   is preferable to a full pipeline abort on a transient section failure.

4. **Executor failure falls back to single call** — If `ThreadPoolExecutor` itself
   raises an unexpected error, `_parallel_report_sections` falls back to the
   existing `_run_with_timeout(ReportWriterAgent(), ...)` approach. This preserves
   the old behaviour as a safety net.

5. **Logging events added** — `report_parallel_start` and `report_parallel_complete`
   events are written to the run log, including the elapsed seconds for Stage 7.

---

## Consequences

Positive:
- On live Vertex AI calls, Stage 7 wall-clock time drops from ~30s to ~10-12s
  (sum→max of section latencies) when all 5 sections complete in parallel
- Individual section failures no longer abort the entire pipeline
- Concurrency is capped at 3 to respect Vertex AI quota defaults

Negative:
- In dry-run mode the fixture read is synchronous and near-instant, so the
  parallel path shows no timing improvement (expected and acceptable)
- Five separate model calls consume slightly more tokens than one consolidated
  call with shared context — estimated overhead is ~200 prompt tokens per run

---

## Files changed

- `agents/report_writer.py` — add `generate_section(section_name, analysis)` method
- `orchestrator/pipeline.py` — add `_parallel_report_sections` helper; replace
  Stage 7 single-call with parallel path + logging
- `tests/test_pipeline.py` — add tests for parallel path and per-section failure tolerance
